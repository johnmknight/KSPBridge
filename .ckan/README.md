# CKAN submission

This directory holds metadata for getting KSPBridge listed on
[CKAN](https://github.com/KSP-CKAN/CKAN), the Comprehensive Kerbal
Archive Network — KSP's de-facto package manager.

## What's here

- `KSPBridge.netkan` — the **netkan** indexer file. It tells the CKAN
  bot how to discover new releases (via GitHub) and how to install
  them (look for the `KSPBridge/` folder in the release zip and copy
  it under `GameData/`).

## Three submission paths (easiest first)

### 1. SpaceDock checkbox (recommended)

When you create the SpaceDock listing for KSPBridge, the new-mod form
has an **"Add mod to CKAN"** checkbox. Tick it. SpaceDock pings the
CKAN team and they index the mod automatically — no separate PR or
issue required. This is the recommended path for most mod authors per
the [official CKAN guide](https://github.com/KSP-CKAN/CKAN/wiki/Adding-a-mod-to-the-CKAN).

### 2. Open an issue at KSP-CKAN/NetKAN

If you skip SpaceDock or just want to ask the CKAN team to do the
work: [open an issue](https://github.com/KSP-CKAN/NetKAN/issues) on
the NetKAN repo with mod details. The CKAN team writes the netkan and
merges it themselves.

### 3. Submit a PR with the netkan yourself

If you want the netkan to be exactly what's in this directory:

1. Fork [`KSP-CKAN/NetKAN`](https://github.com/KSP-CKAN/NetKAN) (NOT
   `CKAN-meta` — that one's auto-populated by the bot and explicitly
   refuses pull requests).
2. Copy `.ckan/KSPBridge.netkan` from this repo into
   `NetKAN/KSPBridge.netkan` of the fork.
3. Open a pull request against
   [`KSP-CKAN/NetKAN`](https://github.com/KSP-CKAN/NetKAN).
4. The CKAN team's CI runs validation; on a green run, a maintainer
   reviews and merges. The CKAN inflater bot then auto-publishes new
   GitHub releases of KSPBridge with no further action.

## Maintenance after listing

The `$kref` (`#/ckan/github/johnmknight/KSPBridge`) tells CKAN to
watch this repo's GitHub releases for new tags. The `$vref`
(`#/ckan/ksp-avc`) tells it to read
`GameData/KSPBridge/KSPBridge.version` from each release zip to
determine version + KSP compatibility — that file is the single
source of truth, regenerated by `scripts/make-release.ps1` from
csproj's `<Version>` on every release. The netkan needs no further
hand-editing for routine version bumps.

If KSP compatibility changes (a future KSP 1.13 etc.), update the
`KSP_VERSION_MIN` / `KSP_VERSION_MAX` literals in
`scripts/make-release.ps1`'s JSON template AND the netkan's
`ksp_version_min` / `ksp_version_max` fields, then PR the netkan
change back to `KSP-CKAN/NetKAN` (path 3 above).
